# StrangeMap 지금까지 한 작업들

작성일: 2026-04-23

## 실행 방식

현재 데모는 프로젝트 루트에서 로컬 프록시 서버를 실행한 뒤 브라우저로 접속한다.

```bash
python3 dev_server.py
```

접속 주소:

```text
http://localhost:3000/subway-route-demo.html
```

`file://`로 직접 열면 네이버 지도 SDK 인증, 장소 검색, 서울시 API 프록시가 정상 동작하지 않을 수 있다.

## 현재 목표

출발지와 목적지를 입력하면 서울 전역 기준으로 대중교통 경로를 찾고, 지도 위에 실제 이동 동선을 표시하는 길찾기 데모를 만드는 중이다.

핵심 방향은 다음과 같다.

- 출발지/목적지 입력을 주소뿐 아니라 장소명 검색 방식으로 처리
- 버스, 지하철, 버스+지하철 경로를 함께 조회
- 가장 빠른 경로와 혼잡도가 낮은 대안을 추천
- 실시간 도착정보와 열차 위치정보를 경로 단계에 연결
- 과거 승하차 데이터를 이용해 현재 혼잡도를 추정
- 선택한 경로를 지도 위에 노선 형태로 표시

## 구현 완료

### 1. 로컬 프록시 서버

`dev_server.py`에서 정적 파일 서빙과 외부 API 프록시를 처리한다.

구현된 주요 프록시:

- `/api/geocode`
- `/api/transit/mixed-route`
- `/api/transit/subway-route`
- `/api/transit/bus-route`
- `/api/subway/realtime`
- `/api/bus/realtime`
- `/api/subway/congestion`
- `/api/bus/congestion`
- `/api/bus/segment-shape`
- `/api/subway/segment-shape`

외부 API를 브라우저에서 직접 호출하지 않고 서버에서 대신 호출하게 구성했다. 이 방식으로 CORS, 인증키 노출, 로컬 테스트 문제를 줄였다.

### 2. 네이버 지도 연동

네이버 Dynamic Map을 연결해 `subway-route-demo.html`에서 지도를 표시한다.

현재 지도 기능:

- 출발지/목적지 마커 표시
- 선택한 대중교통 경로 표시
- 도보 구간은 회색 선으로 표시
- 버스 구간은 파란색 선으로 표시
- 지하철 구간은 호선 색상에 맞춰 표시
- 경로 선택 시 지도 위 동선 갱신

### 3. 장소 검색과 좌표 변환

입력값이 너무 빡빡하게 매칭되는 문제를 줄이기 위해 장소 검색 기반 후보 추천을 추가했다.

현재 검색 흐름:

- 네이버 Local Search API로 장소 후보 조회
- 네이버 Geocoding API로 주소 좌표 변환
- 서울시 대중교통 `getLocationInfo` fallback
- OpenStreetMap Nominatim fallback

개인 상가나 장소명처럼 일반 주소 변환에서 실패하던 입력도 네이버 장소 검색 후보로 보완할 수 있게 했다.

### 4. 대중교통 경로 조회

서울특별시 대중교통환승경로 조회 서비스를 이용해 실제 경로 후보를 받아온다.

사용 중인 기능:

- `getPathInfoByBusNSub`
- `getPathInfoBySubway`
- `getPathInfoByBus`

현재 UI에서는 먼저 전체 후보를 조회한 뒤 필터링한다.

- 전체 최적 경로
- 지하철만
- 버스만
- 버스 + 지하철

초기에는 필터 선택에 따라 API 자체를 다르게 호출하려 했지만, 지금은 사용자가 말한 방향대로 모든 후보를 먼저 가져오고 UI에서 조건별로 걸러 보여주는 구조로 바꿨다.

### 5. 경로 추천 정리

추천 대안은 단순 목록이 아니라 목적별로 정리한다.

- 가장 빠른 경로: API 소요시간이 가장 짧은 경로
- 가장 원활한 경로: 혼잡도 점수가 가장 낮은 경로
- 빠른 원활 대안: 혼잡도도 낮고 두 번째 대안보다 가능한 한 빠른 경로

소요시간은 현재 서울시 경로 API가 내려주는 총 소요시간을 우선 반영한다. 도보/버스/지하철 단계별 시간은 API 총 시간을 기준으로 구간 비중에 따라 나눠 표시한다.

### 6. 실시간 정보

지하철 실시간 도착정보와 실시간 열차 위치정보를 연결했다.

경로에 지하철 구간이 있으면 다음 정보를 표시한다.

- 탑승역 기준 도착 예정 열차
- 환승역 기준 도착 예정 열차
- 현재 열차 위치
- 열차 번호
- 방면 정보

버스 실시간 도착정보도 구조는 만들어져 있다. 다만 버스 도착정보 API는 서비스 권한이나 인증키 승인 상태에 따라 실패할 수 있어 UI에서 실패 메시지를 처리하도록 했다.

### 7. 혼잡도 기능

서울시 승하차 인원 데이터를 이용해 혼잡도를 추정하는 기능을 추가했다.

지하철 데이터:

- 서울시 지하철호선별 역별 승하차 인원 정보
- 서울시 지하철 호선별 역별 시간대별 승하차 인원 정보

버스 데이터:

- 서울시 버스노선별 정류장별 승하차 인원 정보
- 서울시 버스노선별 정류장별 시간대별 승하차 인원 정보

현재 혼잡도 로직은 과거 시간대별 승하차 데이터를 기반으로 현재 시간대의 승객 규모를 추정한다.

기본 계산 방향:

- 현재 시간대의 승차/하차 인원 합산
- 해당 노선 또는 호선의 시간당 운행 빈도 추정
- `시간대 예상 승객 수 / 시간당 예상 운행 횟수`로 차량 또는 노선 단위 혼잡도 추정
- 점수는 0~100으로 정규화

UI 기준:

- 25 이하: 원활, 파랑
- 50 이하: 보통, 초록
- 75 이하: 약간 혼잡, 주황
- 100 이하: 혼잡, 빨강

### 8. 지도 위 노선 표시

사용자가 요청한 것처럼 단순 직선 연결이 아니라 노선 형태에 가깝게 표시하도록 개선했다.

버스:

- 서울 버스 노선 API의 `getRoutePath`를 사용
- 전체 노선 좌표 중 출발 정류장과 도착 정류장에 가까운 구간만 잘라 표시
- 도로를 따라가는 실제 버스 노선에 가까운 형태로 표시

지하철:

- 서울시 경로 API의 지하철 링크 정보만으로는 실제 선형 좌표가 부족하다.
- 현재는 호선별 역 목록과 역 좌표를 조합해 역 순서대로 이어서 표시한다.
- 실제 터널 선형까지는 아니지만, 어느 역들을 지나가는지에 맞춘 노선형 표시다.

## 현재 한계

- 지하철 노선 표시가 실제 선로 좌표가 아니라 역 좌표 기반 근사다.
- 버스 실시간 도착정보는 인증 권한 상태에 따라 실패할 수 있다.
- 경로 API가 세부 구간별 시간을 모두 명확하게 주지 않아, 단계별 시간은 총 소요시간을 기준으로 추정한다.
- 혼잡도는 실시간 탑승 인원 데이터가 아니라 과거 승하차 통계를 이용한 추정값이다.
- 버스 노선은 실제 도로 선형에 가깝지만, 회차나 순환 노선에서는 구간 자르기가 완벽하지 않을 수 있다.
- 일부 장소 검색은 네이버 Local Search 결과 품질에 영향을 받는다.

## 2026-04-28 추가 작업 내역

### 9. 공식 지하철 혼잡도 API 연동

기존의 지하철 혼잡도 추정 로직 위에 서울 열린데이터광장의 공식 지하철 혼잡도 API를 우선 연결했다.

사용한 서비스:

- `subwConfusion`

현재 방식:

- 역명과 호선명을 기준으로 공식 혼잡도 API 조회
- 현재 요일과 현재 30분 구간에 맞는 값 선택
- 공식 API 값이 있으면 그 값을 우선 사용
- 없을 때만 기존 승하차 기반 추정 로직 fallback

변경점:

- 혼잡도를 더 이상 항상 추정하지 않고, 가능하면 공식 퍼센트 값을 그대로 사용
- 100% 초과 혼잡도도 표시 가능하도록 프론트 라벨 기준을 확장
- 예: `126.3%` 같은 값도 `매우 혼잡`으로 렌더링 가능

### 10. 지하철 혼잡도 방향 매칭

초기 공식 혼잡도 연동에서는 같은 역의 상행/하행 또는 내선/외선 값을 평균 내던 문제가 있었다.

현재는 경로 구간 정보를 이용해 방향을 추론해 해당 방향 한 줄만 사용한다.

입력으로 활용하는 값:

- `fromId`
- `toId`
- `fromName`
- `toName`
- `railLinkCount`

현재 방향 처리:

- 2호선: `내선` / `외선`
- 그 외 일반 노선: `상선` / `하선`

효과:

- `잠실역 -> 강남역` 같은 구간에서 평균이 아니라 실제 이동 방향 기준 혼잡도 사용
- 상세 문구도 `상하선 평균` 대신 선택된 방향 하나만 표시

### 11. 검색 후보 정렬 개선

장소 검색에서 정확한 역명이나 장소명이 있어도 주변 상가/건물이 먼저 뜨는 문제가 있어 후보 정렬 기준을 손봤다.

현재 정렬 기준:

- 입력값과 정확히 일치하는 후보가 있으면 무조건 최상단
- exact match가 없을 때만 점수 기반으로 대안 정렬

점수에 반영하는 요소:

- `naverLocalPlace` 우선
- `naverGeocode` 보조 우선
- 지하철/철도 카테고리 가산점
- 이름 exact/substr 일치
- 주소 substr 일치

효과:

- `신내역`, `상봉역`처럼 역명 검색 시 정확히 일치하는 결과가 있으면 가장 위에 배치

### 12. 역 검색 전용 후보 추가

정렬만으로는 해결되지 않는 경우가 있어, 역명 의도로 보이는 검색에는 서울시 지하철 역 검색 결과를 별도로 넣도록 수정했다.

현재 추가한 흐름:

- 검색어가 `역`을 포함하거나 짧은 역명 형태면 역 검색 의도로 판단
- `SearchInfoBySubwayNameService`로 지하철 역 후보 조회
- 역 좌표를 보정해 `seoulSubwayStation` 소스로 후보 목록에 추가
- 일반 네이버 장소 검색 결과보다 앞에 올 수 있게 결합

추가 효과:

- `상봉역` 검색 시 코스트코, 홈플러스 같은 주변 상가만 뜨던 문제를 완화
- 역 자체가 후보 목록에 직접 들어오도록 개선

### 13. 역 좌표 보정 로직 안전화

역 좌표를 잡을 때 네이버 로컬 검색 결과의 첫 번째 항목을 무조건 쓰던 fallback이 있었는데, 이 때문에 역 대신 상가 좌표가 들어갈 가능성이 있었다.

현재는 다음 조건을 만족하는 경우에만 역 좌표로 인정한다.

- `placeName`이 역명과 일치
- 카테고리에 `지하철`이 있거나 이름에 `역`이 포함

즉, 역이 아닌 일반 상가를 역 좌표로 잘못 쓰는 확률을 줄였다.

### 14. 경로 조회 속도 개선

체감 속도가 느린 원인을 찾아 프론트 흐름을 바꿨다.

기존 병목:

- 경로 화면을 보여주기 전에 추천 대안들의 혼잡도 API를 먼저 전부 조회
- 지도 선형을 구간별로 순차 로딩
- 실시간 정보 조회 단계에서 혼잡도 API를 다시 한 번 중복 호출

현재 변경 내용:

- 기본 경로/요약/단계는 먼저 바로 렌더링
- 혼잡도 보강은 백그라운드에서 비동기로 갱신
- 지도 구간 선형은 병렬 로딩
- 경로 도형, 버스 혼잡도, 지하철 혼잡도에 프론트 캐시 추가
- 선택 경로 렌더링 시 실시간 정보는 별도 비동기 로드

효과:

- 첫 화면이 더 빨리 뜸
- 추천 카드와 실시간 정보는 뒤이어 갱신됨
- 같은 구간 재조회 시 중복 호출 감소

### 15. 네이버 장소 검색 정렬 안정화

네이버 Local Search가 `random` 정렬로 되어 있던 부분을 `comment` 정렬로 변경했다.

효과:

- 같은 검색어를 반복 입력했을 때 후보 순서가 불안정하게 흔들리던 문제 완화

### 16. 현재 상태 요약

현재 기준으로 바뀐 핵심은 다음과 같다.

- 지하철 혼잡도는 가능하면 공식 API 값을 사용
- 지하철 혼잡도는 방향별로 구분해 사용
- 역명 검색 시 역 후보를 더 강하게 우선
- 경로/지도/실시간 정보 로딩 체감 속도 개선

### 17. 남아 있는 보완 포인트

- 일부 분기 노선이나 특수 구간은 방향 추론이 완벽하지 않을 수 있음
- 역명 검색에서 여전히 특정 역이 누락되면 역 전용 좌표 소스 추가 보강 필요
- 서울시 경로 API 자체 응답 시간이 느릴 때는 서버 측 경로 캐시를 추가로 고려할 수 있음
